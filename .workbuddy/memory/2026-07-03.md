## 期货实时价格邮件提醒 Web 应用 — 项目启动

### 背景
将现有桌面版 GUI（`futures_price_email_gui_v2.py`，基于 customtkinter）迁移至 Web 端，支持云服务器 7×24 部署。

### 技术选型决策
- 后端：FastAPI + uvicorn（异步、原生 WebSocket）
- 前端：Vue 3 + Element Plus via CDN（免构建，便于云部署，暗色科技风）
- 数据库：SQLite（stdlib sqlite3）
- 实时推送：FastAPI WebSocket
- 行情：TqSdk 后台 daemon 线程 + wait_update() 循环
- 邮件：smtplib（复用 GUI 版 SSL/TLS 回退逻辑）

### 架构要点
- TqSdk 在独立后台线程运行 wait_update()，不阻塞 FastAPI 事件循环
- 线程间通信：通过 run_coroutine_threadsafe + 队列桥接到 asyncio 事件循环，再经 WebSocket 推送前端
- 配置共享加锁：tracker 线程与 Web 路程共享配置，使用 threading.Lock
- 邮件发送由本地时钟间隔触发（send_on_interval_only + should_send），不持续访问 tqsdk

## Bug 修复：停止服务按钮无效

### 根因（3 个缺陷叠加）
1. `FuturesPriceTracker.stop()` 中 `self._api.close()` 同步阻塞 → 卡死 FastAPI 事件循环
2. `AppState.stop()` 在 `with self._lock:` 内调用 `tracker.stop()` → 与 `_tracker_loop` finally 块争锁，状态更新延迟
3. 前端 `stopService()` 成功后未更新 `running.value` → 完全依赖 WebSocket 推送，事件循环阻塞时 UI 不变

### 修复
1. `tracker.stop()`：`api.close()` 移至独立 daemon 线程执行，不阻塞调用方
2. `AppState.stop()`：锁内仅设 `_running=False`，锁外调 `tracker.stop()` 并广播 `status_change`
3. 前端 `stopService()`：成功后立即 `running.value=false` + `statusText="服务已停止"`

## 邮件去抖动合并（debounce）

### 问题
`send_on_interval_only=false` 模式下，7个合约在不同时刻各自变化触发多轮 wait_update，每轮各发一封邮件，短时间收到多封。

### 解决方案
添加去抖动窗口（DEBOUNCE_WINDOW=3秒）：
- 检测到任一合约变化 → 标记 pending_email=True，设 debounce_deadline=now+3s
- 窗口内的新变化不重置窗口（避免无限延后）
- 窗口结束（now >= debounce_deadline）→ 发一份全量快照，清除 pending

### 两种模式行为
- `send_on_interval_only=true`：定时发全量，不依赖变化（不变）
- `send_on_interval_only=false`：变化触发 + 3秒去抖动合并 → 一份全量

### 保证
无论哪种模式，一封邮件 = 所有合约的一张 HTML 表格

## 诊断：邮件"发送成功"但实际收不到

### 根因
1. **tracker.py 日志误导**：`已发送合并邮件` 在邮件**提交到线程池后立即打印**，不代表 SMTP 真正发出去了
2. **emailer.py SMTP 日志是 DEBUG 级别**：连接/登录/发送全过程日志在终端 INFO 级别下**完全不可见**
3. **_send_batch_email 是 fire-and-forget**：提交到线程池后不检查返回值，发送失败被静默吞掉

### 修复（已完成）
1. `emailer.py`：SMTP 连接/登录/发送步骤日志从 `logger.debug` 升级为 `logger.info`，带 [1/4]~[4/4] 阶段标记
2. `tracker.py` `_send_batch_email`：改为 `future.result(timeout=60)` 同步等待结果，返回 bool
3. `tracker.py` 主循环：根据返回值打印 `✅ 邮件发送成功` 或 `❌ 邮件发送失败`
